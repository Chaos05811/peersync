PeerSync Architecture

● WebRTC / PeerJS ● RGA CRDT ● Chunk Transfer ● Star Topology ● SHA-256 Integrity
System Design

Architecture Overview

PeerSync uses a star mesh: a host peer coordinates operations, while all data is transmitted peer-to-peer over WebRTC data channels. No application server is ever involved at runtime.

Presentation Layer

Browser UI — index.html + styles.css

8-zone layout (Title · Share · Menubar · Toolbar · Sidebar · Canvas · Peer Panel · Statusbar). Pure HTML/CSS — no framework. contenteditable editor with caret-preserving DOM patch on every CRDT update.

CRDT Engine

RGA — Replicated Growable Array

Each character is a node with a globally unique peerId:clock ID. Concurrent inserts tie-break deterministically. Deletes are tombstoned. All peers converge to the same text regardless of message order.

crdt.js
P2P Network

PeerManager

Star topology over PeerJS (WebRTC). Host relays CRDT ops + cursor positions to all spoke peers. Pings every 2 s for live latency display. File transfers route through host relay.

peer-manager.js
↓
Transport

WebRTC RTCDataChannel (via PeerJS 1.5.4)

Reliable ordered data channels (SCTP). STUN: stun.l.google.com:19302. Signaling: PeerJS public cloud (initial handshake only — zero server involvement once peers are connected). JSON serialization over data channel.

Data Flows

Key Message Flows

How data moves through the system for the three primary operations.

📝 Collaborative Edit
User keypress → CollabEditor diff → RGA insert/delete → PeerManager.sendOps() → Host relay → All peers → crdt.applyRemote() → DOM patch + caret restore
📁 File Transfer
file-announce → SHA-256 hash → file-start → 16 KB chunks ×N → file-done → Reassemble Uint8Array → Verify SHA-256 → Blob + download link
🔗 Peer Discovery / Session Join
New peer opens URL#roomId → PeerJS connect(roomId) → send {type:'join'} → Host sends {type:'sync', crdtState} → crdt.deserialize() → Editor renders doc
Execution Plan

Build & Run

PeerSync is a fully static site. No build step, no backend, no package manager needed.

01

Serve the static files

Any static file server works. Examples:

npx serve .  ·  python3 -m http.server 8080  ·  Nginx / Caddy / GitHub Pages / Netlify Drop

No build step
02

Open in browser — peer connects to PeerJS cloud

PeerJS public signaling server handles the initial WebRTC handshake. Once the RTCDataChannel is open, no further server communication occurs.

Signaling only — no data passes through server
03

Click "New Document" to host

Your PeerJS peer.id becomes the room code. Share it or the URL hash with collaborators.

04

Collaborators paste the room code → Join

They connect to you via WebRTC. You push the current crdt.serialize() state so they start with the same document. All subsequent edits travel as individual RGA ops.

05

Attach a file — it travels P2P

The host splits the file into 16 KB chunks and relays them to all peers. Recipients reassemble and verify SHA-256. A download link appears when transfer is complete.

Tech Stack

Technology Choices

Transport

WebRTC RTCDataChannel

Browser-native P2P. Reliable SCTP. No server roundtrips after handshake.

Signaling

PeerJS 1.5.4 (CDN)

Abstracts ICE/SDP negotiation. Free public server used for initial handshake only.

Consistency

RGA CRDT

Replicated Growable Array. Op-based: every insert/delete is self-describing and commutative.

Integrity

SubtleCrypto SHA-256

Browser-native hashing. File digest computed by sender; verified by each receiver after reassembly.

Frontend

Vanilla JS + HTML

Zero build step, zero framework. 8 files, loads in < 300 ms.

Fonts

DM Sans + JetBrains Mono

UI text + monospace code/status elements. Google Fonts CDN.

State

localStorage

Recent document history and last document content persisted locally per browser.

File Transfer

16 KB Chunk Pipeline

10-chunk batches with 15 ms yield — prevents RTCDataChannel backpressure without flow-control API.

Known Limitations

Current Constraints

!

Host is a SPOF

If the host peer disconnects, new edits can no longer be relayed. A host-election protocol (Bully or Ring) can be added to promote the peer with the lowest ID. For now, the document is frozen and users are notified.

!

JSON serialization overhead

PeerJS serializes data as JSON. Large documents or high-frequency edits can approach the 256 KB message limit. A binary encoding (MessagePack / CBOR) would reduce overhead by ~40%.

!

CRDT tombstone growth

Deleted characters remain in the RGA node list indefinitely. A compaction pass (garbage-collecting tombstones once all peers have applied the deletion) would be needed for very long-lived documents.

!

PeerJS signaling dependency

First connection requires the PeerJS public server. Replace with a self-hosted peerjs-server instance (npm i -g peerjs) to remove this external dependency.
